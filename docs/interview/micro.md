## **1. 什么是微前端？为什么要做微前端？**
### **标准回答**
微前端（Micro Frontends）是一种将大型前端单体应用拆分成多个可独立开发、独立部署、独立运行子应用的架构模式，其思想来源于后端微服务。

它的核心目标不是“拆项目”，而是解决大型前端工程在多人协作、技术演进和部署效率上的问题。

比如一个大型中后台系统，可以拆成：

+  用户中心 
+  权限中心 
+  商品中心 
+  订单中心 
+  营销中心 

每个业务模块由不同团队独立维护，彼此低耦合。

### **微前端解决的问题**
在大型项目中，传统单体前端会遇到几个明显问题：

#### **（1）多人协作冲突严重**
几十名前端同时开发一个仓库，容易出现：

+  Git 冲突频繁 
+  发版互相影响 
+  测试范围不可控 

一次发布可能影响全站。

微前端拆分后：

每个子应用独立开发、独立上线。

即使订单系统发布失败，也不会影响用户系统。

---

#### **（2）技术栈升级困难**
例如旧项目使用：

+  AngularJS 
+  Vue2 

但新业务需要：

+  React18 
+  Vue3 

如果是单体架构：

升级成本极高。

而微前端允许：

新模块逐步迁移。

例如：

+  老模块继续运行 Vue2 
+  新模块使用 React18 

实现渐进式升级。

---

#### **（3）构建速度和维护成本问题**
大型项目：

```plain
500+ 页面
几十万行代码
```

每次构建可能十几分钟。

微前端后：

子系统单独构建。

提升 CI/CD 效率。

---

## **2. 微前端有哪些实现方案？**
### **（1）iframe 方案**
最简单直接。

把子系统放进 iframe：

```plain
<iframe src="/user-system"></iframe>
```

优点：

+  完全隔离 
+  技术栈独立 
+  实现简单 

缺点也明显：

+  通信困难 
+  路由管理麻烦 
+  用户体验差 
+  SEO 不友好 

所以：

通常不作为核心业务方案。

更多用于第三方系统接入。

---

### **（2）路由分发式**
通过 Nginx 或网关：

```plain
/user → React 应用
/order → Vue 应用
```

优点：

+  简单稳定 
+  易于落地 

缺点：

不能真正做到应用共存。

本质还是多个 SPA。

---

### **（3）Web Components**
利用浏览器原生能力：

```plain
customElements.define()
```

实现组件隔离。

优点：

+  天然样式隔离 
+  技术栈无关 

缺点：

+  生态成熟度一般 
+  学习成本较高 

企业落地不多。

---

### **（4）JS Entry（主流方案）**
目前企业最常见。

代表框架：

qiankun

核心思想：

主应用动态加载子应用资源。

主应用负责：

+  路由分发 
+  生命周期管理 
+  沙箱隔离 

子应用：

独立运行。

这是目前大型中后台系统最主流的方案。

---

### **（5）Module Federation**
来自：

Webpack 5。

它的核心不是“应用拆分”，而是：

模块共享。

例如：

多个系统共用：

+  React 
+  组件库 
+  工具函数 

避免重复打包。

所以：

qiankun 更偏应用级微前端。  
Module Federation 更偏模块级共享。

很多企业会组合使用。

---

## **3. qiankun 原理是什么？**
### **核心回答**
qiankun 的底层核心：

**single-spa + import-html-entry**

它主要做了四件事：

### **第一：路由劫持**
监听：

+  hashchange 
+  popstate 

当 URL 变化时：

判断当前应该激活哪个子应用。

例如：

```plain
/order/*
```

激活订单系统。

---

### **第二：动态资源加载**
主应用请求子应用 HTML。

解析其中：

```plain
script
link
style
```

然后动态执行 JS。

本质类似：

```plain
import()
```

动态加载资源。

所以：

子应用不需要提前打包进主应用。

---

### **第三：生命周期管理**
每个子应用必须导出：

```plain
bootstrap
mount
unmount
```

作用分别是：

+  bootstrap：初始化 
+  mount：挂载应用 
+  unmount：卸载应用 

主应用负责统一调度。

这也是为什么：

子应用可独立运行，也能嵌入运行。

---

### **第四：沙箱隔离**
避免子应用污染全局环境。

比如：

```plain
window.a = 1
```

不能影响其他子应用。

因此 qiankun 提供：

+  JS 沙箱 
+  CSS 隔离 

---

## **4. qiankun 沙箱隔离原理？**
**这是高频深挖题。**

**qiankun 主要有两种沙箱：**

### **（1）SnapshotSandbox**
快照沙箱。

原理：

进入应用前保存：

```plain
window
```

退出时恢复。

类似：

```plain
Object.assign()
```

优点：简单。

缺点：性能差。

**现在基本不用。**

---

### **（2）ProxySandbox（主流）**
基于：

```plain
Proxy
```

代理：

```plain
window
```

例如：

子应用执行：

```plain
window.test = 1
```

实际上写入的是代理对象。

退出应用：直接销毁代理，不会污染全局。

这是当前主流方案。

---

## **5. 微前端如何做应用通信？**
常见方案有五种。

### **props 传值**
主应用注册时：传递参数。

适合：父子关系简单场景。

---

### **全局状态管理**
qiankun：

```plain
initGlobalState()
```

实现共享数据。

例如：

+  用户信息 
+  Token 
+  权限 

---

### **Event Bus**
例如：

```plain
mitt
```

发布订阅模式。

解耦程度高。

适合：跨应用通知。

---

### **浏览器通信能力**
例如：

```plain
postMessage
BroadcastChannel
```

用于跨上下文通信。

---

### **共享 Store**
如 Redux / Pinia。

一般不推荐。

因为：容易造成强耦合。

---

## **6. 微前端如何共享登录态？**
**高频。**

核心思路：

单点登录（SSO）。

一般做法：用户登录主系统后：

1.主应用获取：token

2.然后：通过 props 或全局状态注入子应用。

同域情况下：Cookie 自动共享。

跨域：通常：JWT + SSO，实现统一认证。

做到：一次登录，全站通用。

---

## **7. qiankun 和 Module Federation 有什么区别？**
**这是经典题。**

**核心区别：**

| 维度 | qiankun | Module Federation |
| --- | --- | --- |
| 粒度 | 应用级 | 模块级 |
| 目标 | 微前端 | 模块共享 |
| 隔离性 | 强 | 相对弱 |
| 部署 | 独立部署 | 联邦部署 |
| 场景 | 中后台拆分 | 组件共享 |


简单说：

qiankun 更关注系统拆分。  
Module Federation 更关注资源共享。

很多大型项目：

qiankun + Module Federation 混合。

例如：

子系统拆分用 qiankun，组件共享用 Module Federation。

---

## **8. 微前端有什么缺点？**
**高分回答一定要讲缺点。**

### **性能问题**
多个子应用：可能加载多个 runtime。

资源增多。首屏变慢。

---

### **通信复杂**
应用间状态同步困难。

容易：数据不一致。

---

### **调试复杂**
问题定位难。

可能涉及：

+  主应用 
+  子应用 
+  沙箱 

排查成本高。

---

### **样式污染**
CSS 冲突：

是高频问题。

需要：

+  CSS Modules 
+  Scoped 
+  Shadow DOM 

隔离。

---

### **运维复杂**
部署链路变长。

CI/CD 更复杂。

---

## **9. 什么情况下不建议用微前端？**
**这个题非常加分。**

**不要无脑说“都适合”。**

**以下场景：**

1. **小团队**

比如：5 人以内。系统不复杂。

微前端：治理成本 > 收益。

2. **强耦合业务**

模块之间：共享状态很多，频繁通信。

拆分意义不大。

3. **中小项目**

几十个页面，单体应用足够，没必要过度设计。

---

## **10. 如果让你落地微前端，你怎么设计？**
**面试高分答案：**

**我会先评估：**

**第一：业务边界**

是否天然拆分：

+  用户中心 
+  权限 
+  订单 

做到：业务自治。

**第二：技术方案**

优先：qiankun

原因：成熟、生态稳定。

**第三：统一规范**

包括：

+  登录态 
+  路由规范 
+  埋点 
+  权限 
+  UI 规范 

否则：容易变成技术灾难。

**第四：性能优化**

例如：

+  prefetch 
+  CDN 
+  按需加载 
+  shared dependency 

减少重复资源。

---

**最后一句总结（适合面试收尾）：**

**微前端不是银弹，它更适合大型组织协作和复杂业务拆分。如果业务规模不足，治理成本可能高于收益，所以技术选型一定是收益、复杂度和团队能力之间的平衡。**

**这个表达会更像高级工程师/架构师。**

# 性能优化
## 一、资源层优化（收益最高）
### 1. 依赖共享，避免重复打包
这是微前端最重要的优化之一。

问题在于：

多个子应用可能都依赖：

```plain
react
react-dom
antd
lodash
```

如果每个应用都独立打包：

同一个依赖会被重复下载。

例如：

三个 React 子应用：

```plain
react × 3
react-dom × 3
```

首屏资源体积会明显增大。

### 优化方案
#### （1）externals + CDN
把公共依赖抽离：

```plain
externals: {
  react: 'React',
  'react-dom': 'ReactDOM'
}
```

由主应用统一加载：

```plain
<script src="react.production.min.js"></script>
```

这样：

所有子应用共享同一个 React 实例。

优点：

+  降低 bundle 体积 
+  加快加载速度 

缺点：

+  依赖版本管理较复杂 

---

#### （2）Module Federation（推荐）
基于：

Webpack 5。

实现运行时依赖共享：

```plain
shared: {
  react: {
    singleton: true
  },
  "react-dom": {
    singleton: true
  }
}
```

核心思想：

多个应用共享同一份依赖。

避免重复下载与重复初始化。

相比 CDN：

更适合大型项目。

---

### 2. 静态资源缓存
必须开启：

+  强缓存 
+  协商缓存 

例如：

```plain
Cache-Control: max-age=31536000
```

同时：

资源 hash 化：

```plain
main.a123.js
```

这样：

用户二次访问几乎不需要重新下载静态资源。

---

### 3. 资源压缩
开启：

```plain
gzip
brotli
```

通常：

JS 体积可减少 50% 以上。

这是成本最低、收益最高的优化方式之一。

---

## 二、子应用加载优化
### 1. 预加载（Prefetch）
以：

qiankun 为例：

支持：

```plain
start({
  prefetch: true
})
```

原理：

用户进入主系统后：

浏览器空闲时提前下载高频子应用资源。

例如：

用户访问：

```plain
/user
```

后台提前加载：

```plain
/order
/product
```

用户切换页面时：

几乎秒开。

但要注意：

不要全部预加载。

否则：

首屏压力过大。

建议：

只预加载高频业务模块。

---

### 2. 路由级懒加载
不要登录后：

```plain
一次加载所有子应用
```

正确做法：

按路由动态加载。

例如：

只有访问：

```plain
/order
```

时：

才 mount 订单系统。

这样能显著降低：

首屏加载时间。

---

### 3. 子应用缓存（KeepAlive）
微前端切换页面：

通常会：

```plain
mount → unmount → remount
```

问题：

+  页面闪烁 
+  接口重复请求 
+  表单状态丢失 

优化方案：

缓存子应用状态。

类似：

+  Vue KeepAlive 
+  React 页面缓存 

适用于：

+  查询页 
+  工作台 
+  长表单 

体验提升非常明显。

---

## 三、运行时优化
### 1. 减少重复初始化请求
很多子应用进入页面都会请求：

```plain
用户信息
权限
菜单
字典
国际化配置
```

问题：

每个应用重复请求。

优化：

由主应用统一管理。

例如：

通过 props 或全局状态：

```plain
props.userInfo
props.permission
```

共享给子应用。

这样：

减少重复 IO 请求。

提升加载速度。

---

### 2. 减少沙箱开销
例如：

qiankun 使用：

```plain
ProxySandbox
```

对：

```plain
window
```

做代理。

如果大量：

```plain
window.xxx
```

访问：

会增加运行时开销。

优化：

尽量使用模块内部状态，不滥用全局对象。

---

### 3. 防止内存泄漏
子应用销毁时：

必须清理：

```plain
eventListener
timer
websocket
observer
```

否则：

频繁切换后：

页面越来越卡。

例如：

React：

```plain
useEffect(() => {
  const timer = setInterval(() => {})

  return () => clearInterval(timer)
}, [])
```

---

## 四、渲染层优化
### 1. 减少无效渲染
React：

```plain
memo
useMemo
useCallback
```

Vue：

```plain
v-memo
```

目标：

避免整个子应用反复 render。

---

### 2. 长列表虚拟化
例如：

1000 条表格数据。

不要全部渲染。

使用：

+  Virtual List 
+  Windowing 

只渲染：

可视区域 DOM。

性能提升明显。

---

### 3. 骨架屏与 Loading
用户感知优化。

子应用 mount 前：

显示：

```plain
Skeleton
Loading
```

减少：

白屏感知时间。

---

## 五、架构层优化（高阶）
### 1. 控制拆分粒度
微前端不是拆得越细越好。

错误：

```plain
一个菜单一个应用
```

会导致：

+  通信复杂 
+  切换成本高 
+  性能下降 

建议：

按业务域拆分：

```plain
订单中心
用户中心
营销中心
```

做到：

高内聚、低耦合。

---

### 2. 主应用提供统一能力
主应用统一管理：

+  登录态 
+  权限 
+  埋点 
+  国际化 
+  UI SDK 

避免：

子应用重复建设。

---

### 3. SSR 优化首屏
对于首屏敏感业务：

可以结合：

Next.js

做 SSR。

减少：

首屏白屏时间。

尤其：

ToC 场景收益明显。

---

## 面试收尾总结（推荐背诵）
微前端性能优化我通常会从五个层面考虑：资源层避免依赖重复加载，比如通过 Module Federation 做共享；加载层通过 prefetch、懒加载和应用缓存减少切换成本；运行时减少重复初始化和沙箱开销，同时避免内存泄漏；渲染层通过 memo、虚拟列表减少渲染压力；最后在架构层控制拆分粒度，避免过度微前端化。本质上微前端优化的核心是减少重复加载、降低初始化成本以及控制运行时复杂度。
